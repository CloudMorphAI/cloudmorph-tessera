# v0.3.0 — Stripe-backed Tier Enforcement (Consumer Side)

## Problem

The OSS package consumes a `tier` claim from a license JWT issued by the license server in `cloudmorph-mono-repo`. The license server is the source-of-truth for "which tier does this customer have." In v0.2.0 the license server's tier value is mostly hard-coded for early customers and not yet tied to Stripe subscription state. The Stripe integration that closes this loop lives in cloudmorph-mono-repo (out of scope for this repo).

What's in scope for the OSS package is the **consumer side** of Stripe-driven tier changes: how does this repo behave when a customer's subscription state changes mid-flight?

The relevant behaviors:

1. **Tier upgrades** propagate cleanly today — the next `LicenseValidator.check()` refresh picks up the new tier from the license server's response.
2. **Tier downgrades** have a cache-coherency question: an `enterprise`-tier customer who downgrades to `developer` still has the enterprise packs in their local cache. Should those continue to load, or should they be invalidated?
3. **Mid-subscription expiry / non-payment** triggers tier=`free` at the license server. The OSS package needs to handle this gracefully — no crashes, no silent data loss, clear logs.

## Scope

This improvement does **not** introduce direct Stripe API calls in the OSS package. The OSS package treats the license JWT's `tier` claim as authoritative; Stripe is a hop further upstream. The work here is consumer-side cache and refresh policy.

### 1. Cache eviction policy on tier change

When `LicenseValidator.check()` returns a tier different from the previous cached value, the in-memory and on-disk caches need treatment:

- **In-memory** — the `LicenseValidator._in_memory_cache` is replaced immediately. The `IntelligenceClient` sees the new tier on next refresh.
- **Pack cache on disk** — packs already extracted under `cache_dir/packs/` are kept. Subsequent CDN fetches for above-tier packs return 403 from the CloudFront tier gate (the producer-side enforcement). The local files continue to exist and continue to be loaded by `policy/loader.py` until they're explicitly deleted.

The design decision: **keep cached packs through tier transitions**. Rationale:

- The customer paid for the higher tier at some point. Keeping the pack means a downgrade-then-re-upgrade cycle doesn't require a fresh download.
- Forcing eviction on downgrade is operator-hostile: a billing failure that takes a day to resolve shouldn't blow away the operator's policy set.
- Local disk content is not the security boundary; the CDN's edge gate is. Hold-over packs cannot fetch new content past their cached version.

The downside: an operator who downgrades intentionally (e.g., enterprise pilot → developer paid plan) and wants to drop the enterprise-tier policies has to either re-run `tessera init` (which wipes the cache and re-fetches) or manually delete the relevant `cache_dir/packs/<name>/` directories. This is an explicit operator action, not an automatic one.

If the founder later prefers strict eviction (delete pack content when its `min_tier > current_tier`), it's an additional flag — `intelligence.evict_above_tier_packs: true` (default `false`). Out of v0.3.0 default scope; recap as a v0.4.0 candidate if any customer hits the corner case.

### 2. License-server response timeout + retry

The current `LicenseValidator.check()` uses a 10-second `httpx.AsyncClient` timeout. On license-server unreachability it falls back to disk cache (up to 7 days old by default). Verify the timeout is right for the production license server's typical latency; consider lowering to 3–5 seconds if 10s is excessive. Out of v0.3.0 scope unless production observation shows the longer timeout is masking issues.

### 3. Logging hygiene around tier transitions

Add three new log events:

- `event=license_tier_change from=<old> to=<new>` — on every detected tier change.
- `event=pack_above_tier_skipped name=<pack> min_tier=<x> current_tier=<y>` — every time `IntelligenceClient._tier_allowed` returns false during a refresh.
- `event=license_subscription_expired customer_id=<id>` — when `LicenseStatus.tier="free"` comes back from the license server and the previous status was a paid tier.

These let operators reconstruct subscription lifecycle from logs without server-side billing data.

### 4. Audit-event tier claim

Every audit event emitted by `AuditEmitter` already carries `pricingSnapshotId` (when Infracost is configured). Extend the emitter to optionally carry `licenseTier` from `LicenseValidator._in_memory_cache.tier` — so post-hoc audit-log analysis can correlate decisions with the customer's tier at the time of the call. The field is optional; absence means the OSS package wasn't running with intelligence enabled.

## What this is NOT

- **Not direct Stripe API calls from the OSS package.** Stripe credentials remain server-side only.
- **Not seat-count enforcement.** That's the admin console's job in `cloudmorph-mono-repo`.
- **Not webhook handling.** Stripe webhooks fire against the license server, not against Tessera.
- **Not billing-data exposure.** The OSS package sees only the JWT claims (`tier`, `exp`, `seats`, `customer_id`); it does not see prices, payment methods, or invoice state.

## Acceptance criteria

- A test that mocks the license server returning `tier: developer` then `tier: free` on subsequent refreshes confirms the `IntelligenceClient` accepts the transition without crashing, logs the tier change, and continues serving cached packs.
- The eviction policy is documented at `status/intelligence-and-licensing.md`: cached packs persist across tier transitions; CDN gating prevents new fetches of above-tier packs.
- Audit events carry `licenseTier` when intelligence is enabled.
- The three new log events appear in correctly-formatted structured log lines.

## Out of scope

- Acceptance of webhooks directly into the OSS package.
- Pro-rata refund handling, dunning, billing portal redirects.
- Multi-account-per-customer subscription handling. One JWT = one tier; complex billing structures are surfaced via the admin console issuing the right JWT per deployment.
