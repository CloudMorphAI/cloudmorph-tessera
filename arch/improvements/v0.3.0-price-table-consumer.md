# v0.3.0 — Consume the Signed Price-Table Artifact

## Problem

The consumer side of the Option-A cost-resolution architecture (described in `status/integrations-and-cost.md`) is unimplemented in v0.2.0. Today the `predicted_cost` condition consults a live Infracost GraphQL backend via `tessera/cost/infracost.py:InfracostClient` — 200ms timeout per call, 300s cache, fail-closed on every error. This works but has four drawbacks:

1. **Latency** — 100–300ms typical Infracost call inside the hot path.
2. **External dependency** — every customer needs network egress to a self-hosted Infracost container or to Infracost's public API.
3. **Rate limits** — Infracost's free tier rate-limits aggressively.
4. **Offline operation** — air-gapped customer environments can't use the firewall's cost conditions at all.

The target architecture is build-time price materialization: the producer (`tessera-intelligence`) runs every mapping YAML's `infracost_query` once per release, materializes results into a signed `aws-prices-<version>.json` artifact, ships it alongside the mapping bundle. The consumer (this repo) fetches the artifact at refresh time, verifies the signature, builds an in-memory lookup keyed `(operation, args_signature, region)`. Lookups are sub-millisecond at MCP call time. No external dependency. Offline-friendly.

The producer-side work is captured in `tessera-intelligence/arch/improvements/v0.3.0-price-table-materialization.md`. This file is the consumer-side counterpart.

## Scope

### 1. Price-table loader at `tessera/cost/price_table.py`

A new module that:

- Loads `aws-prices-<version>.json` from `~/.tessera/intelligence/mappings/<bundle>/<version>/` at `IntelligenceClient.refresh()` time.
- Ed25519-verifies the artifact's `signature` field against the bundled public key (using the existing `tessera/intelligence/public_key.pem` machinery — same key, same canonical-JSON canonicalization as packs).
- Parses the `entries` dict into an in-memory `dict[str, PriceTableEntry]` keyed on the canonical key (`<operation>|<args_signature>|<region>`).
- Exposes `cost_for_call(operation, args, region) → float | None` API. Returns USD cost in <1ms. Returns `None` for unmapped operations.
- Handles ceiling-band entries by reading `price_realm: per_token` and multiplying by `args.maxTokens` at call time.

### 2. Wire the loader into the proxy lifespan

In `tessera/proxy.py:_lifespan`, after the `IntelligenceClient` refresh completes, scan `client.get_cached_mappings_dir()` for `aws-prices-*.json` artifacts. Load each, verify, hold the resulting `PriceTable` instance on `app.state.price_table`. The proxy's pre-fetch step (currently calling `InfracostClient.query_sku`) consults the price table first; falls back to Infracost only if the operation is in the mapping but not in the price table (e.g., during a window where mapping ships but price-table hasn't materialized yet).

### 3. Engine wiring

The `predicted_cost` condition evaluator at `tessera/policy/conditions.py:_evaluate_predicted_cost` already reads from `context["cost_cache"]`. The pre-fetch step in the proxy populates this cache. The new code path is: pre-fetch consults `price_table.cost_for_call()` first; on `None` falls back to `cost_backend.query_sku()`. No condition-evaluator changes are required — the contract through `context["cost_cache"][tool_name]` is unchanged.

### 4. Azure-prices parity

When v1.1.0 Azure mappings ship from the producer side (`tessera-intelligence/arch/improvements/v1.1.0-azure-mappings.md`), the `azure-prices-<version>.json` artifact uses the identical format. The price-table loader is provider-agnostic: it scans for any `*-prices-*.json` in the mappings cache, verifies each, merges into a single in-memory lookup. AWS and Azure operations coexist in one table since their canonical keys include the operation name (which is provider-prefixed: `aws_*` vs `azure_*`).

### 5. Removal of Infracost as the primary path

When the price table is loaded for the requested operation, `InfracostClient` is not consulted. Infracost remains as the fallback for operations in mappings but not yet in the price table (a transitional state during materializer development). Once the producer-side v0.3.0 ships and the price table covers the full mapping surface, Infracost is dormant in normal operation. Whether to deprecate it entirely is a v0.4.0 decision; for v0.3.0 it stays as fallback.

## Price-table file format (consumer side)

The consumer expects this shape, signed with the same Ed25519 key as packs:

```jsonc
{
  "$schema": "https://tessera-intelligence.cloudmorph.io/schemas/price-table.schema.json",
  "provider": "aws",
  "version": "v1.0.0",
  "generated_at": "<RFC 3339>",
  "infracost_snapshot_id": "<opaque>",
  "entries": {
    "aws_ec2_RunInstances|InstanceType=m5.large|us-east-1": {
      "unit_cost_usd": 0.096,
      "currency": "USD",
      "price_realm": "instance_hour"
    },
    "aws_bedrock_InvokeModel|modelId=anthropic.claude-opus-4-7|us-east-1": {
      "unit_cost_usd": 0.000015,
      "currency": "USD",
      "price_realm": "per_token"
    },
    ...
  },
  "content_hash": "<sha256 hex>",
  "signature": "<ed25519 base64>"
}
```

Key format: `<operation>|<args_signature>|<region>`. The args signature is the deterministic canonical encoding of the pricing-relevant args (InstanceType for EC2 RunInstances, DBInstanceClass + Engine + MultiAZ for RDS, modelId for Bedrock, etc.). Irrelevant args are omitted.

`price_realm` enumerates how the consumer interprets `unit_cost_usd`:

- `instance_hour` — per-hour rate; multiply by expected runtime hours.
- `gb_month` — per-GB-month rate; multiply by storage size and duration.
- `request` — per-request rate.
- `invocation` — per-invocation rate.
- `per_token` — per-token rate; multiply by `maxTokens` from call args (the Bedrock ceiling-band case).
- `fixed_monthly` — flat monthly cost regardless of usage.

The consumer's `cost_for_call(operation, args, region)` returns the multiplied result for the call-shape inputs, not the raw unit rate.

## Acceptance criteria

- `aws-prices-<version>.json` shipped by the producer loads at `IntelligenceClient.refresh()` time without error.
- Signature verification against the bundled public key succeeds; a tampered artifact (one byte changed) fails verification and logs a clear error.
- `cost_for_call(operation, args, region)` returns sub-millisecond on all 37 mapped operations.
- `predicted_cost` conditions in policies make decisions byte-identical to today's Infracost-backend path, where the price-table data agrees with Infracost's response.
- Ceiling-band (Bedrock) per-token rates are correctly multiplied by `args.maxTokens` to produce a ceiling estimate.
- A negative test confirms `cost_for_call` for an unmapped operation returns `None` (does not raise).
- The proxy lifespan does not start if a present price-table file fails signature verification (fail-loud on tamper).

## Out of scope

- **Tampering with `aws_mapping.py` builtins.** The 10 builtin mapping functions stay; they're the fallback when the price table is absent (e.g., a freshly-installed package with no intelligence cache yet). The price table augments, doesn't replace, the mapping layer.
- **Per-customer reserved-instance discounts.** Price table reflects public on-demand pricing; customer-specific RI/SP discounts are applied (if at all) downstream in the customer's own cost-model layer.
- **Live Infracost deprecation.** Infracost remains the fallback for unmapped operations and for diagnostic scenarios where an operator wants to compare price-table data against live pricing.
