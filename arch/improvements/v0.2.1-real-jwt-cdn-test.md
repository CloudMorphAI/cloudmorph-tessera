# v0.2.1 — Consumer-side End-to-End Real-JWT CDN Test

## Problem

The CloudFront license-tier gate has been partially verified — anonymous fetch correctly returns 401 (confirmed at v0.2.0 ship via `curl` with no `X-Tessera-License` header). The seven remaining scenarios in the verification matrix (in-tier 200s, above-tier 403s across `developer` / `scale` / `enterprise` tiers) are unverified end-to-end against the production CDN.

This is the consumer-side counterpart to the identically-named improvement in tessera-intelligence (`tessera-intelligence/arch/improvements/v0.2.1-real-jwt-cdn-test.md`). From this repo's POV, the test exercises `IntelligenceFetcher` (the `tessera/intelligence/client.py` runtime) with real signed license JWTs across the matrix, asserting that:

- `IntelligenceClient.refresh()` succeeds when the customer's tier is `≥ pack.min_tier`.
- `IntelligenceClient.refresh()` skips above-tier packs without crashing; the failed pack is recorded in the `errors` return field rather than raising.
- The bundled `tessera/intelligence/public_key.pem` correctly verifies signatures on returned content.
- The `LicenseValidator` correctly extracts the `tier` claim from the JWT and feeds it into the manifest filter.

## The 8-scenario matrix

| Scenario | License JWT tier | Pack requested | Expected outcome |
|----------|------------------|----------------|------------------|
| Anonymous | (none) | catalog | 401 ✓ (verified at v0.2.0 ship) |
| developer JWT → catalog | `developer` | `/v1.0.0/catalogs/pack-index.json` | 200 |
| developer JWT → developer pack | `developer` | `aws-cost-aware-defaults` | 200 |
| developer JWT → enterprise pack | `developer` | `hipaa-guardrails` | 403 |
| scale JWT → developer pack | `scale` | `aws-cost-aware-defaults` | 200 |
| scale JWT → fintech pack | `scale` | `fintech-pack` | 200 |
| scale JWT → enterprise pack | `scale` | `hipaa-guardrails` | 403 |
| enterprise JWT → any | `enterprise` | any pack path | 200 |

The first row is verified; the other seven are pending.

## Implementation steps

1. **Issue a test license set** via the admin app at `admin.cloudmorph.io` (in cloudmorph-mono-repo). Issue three licenses for a test tenant — `developer`, `scale`, `enterprise`. Capture the JWTs from the admin UI; they have 24-hour expiry by default which is plenty for the test.

2. **Add `tests/integration_cdn_smoke.py`**, gated on `TESSERA_INTEGRATION_TESTS=1`. The test runs the IntelligenceClient against each tier:

   ```python
   # tests/integration_cdn_smoke.py
   import os
   import pytest
   from tessera.config import IntelligenceConfig
   from tessera.intelligence.client import IntelligenceClient
   from tessera.intelligence.license import LicenseValidator

   pytestmark = pytest.mark.skipif(
       os.environ.get("TESSERA_INTEGRATION_TESTS") != "1",
       reason="set TESSERA_INTEGRATION_TESTS=1 to run live CDN smoke",
   )

   def _client_for_tier(jwt: str) -> IntelligenceClient:
       os.environ["TESSERA_LICENSE_KEY"] = jwt
       cfg = IntelligenceConfig(enabled=True, ...)  # production URLs
       pub_key = (importlib.resources.files("tessera.intelligence") / "public_key.pem").read_bytes()
       validator = LicenseValidator(config=cfg, public_key_pem=pub_key)
       return IntelligenceClient(config=cfg, license_validator=validator)

   @pytest.mark.parametrize("tier_env_var, expected_packs", [
       ("DEVELOPER_JWT", {"vendor-mcp-protection", "aws-cost-aware-defaults"}),
       ("SCALE_JWT",     {"vendor-mcp-protection", "aws-cost-aware-defaults", "fintech-pack"}),
       ("ENTERPRISE_JWT",{"vendor-mcp-protection", "aws-cost-aware-defaults", "fintech-pack",
                          "hipaa-guardrails", "pci-dss-controls", "healthcare-pack"}),
   ])
   async def test_tier_eligible_packs(tier_env_var, expected_packs):
       jwt = os.environ[tier_env_var]
       client = _client_for_tier(jwt)
       result = await client.refresh(force=True)
       cached = {p.parent.name for p in client.get_cached_packs()}
       assert cached >= expected_packs
   ```

3. **Run the matrix**:

   ```bash
   DEVELOPER_JWT=ey... SCALE_JWT=ey... ENTERPRISE_JWT=ey... \
     TESSERA_INTEGRATION_TESTS=1 \
     pytest tests/integration_cdn_smoke.py -v
   ```

4. **For any divergence**, dive into the CloudFront Function — the `min_tier` path-prefix lookup may have a wrong mapping, or the JWT `tier` claim may differ from what the function expects. This is a producer-side investigation; the consumer-side test surfaces the symptom.

5. **Also verify the OSS fetcher's 403 handling** — refreshing with a `developer` JWT against a catalog that includes enterprise packs should record an error in the `refresh()` return dict but not crash. Confirm `client.get_cached_packs()` returns only the tier-eligible packs.

## Acceptance criteria

- All 8 scenarios match expected outcome.
- `tests/integration_cdn_smoke.py` passes locally with `TESSERA_INTEGRATION_TESTS=1` and the three test JWTs in env.
- The integration test is added to CI but gated behind the env var so it doesn't run on every PR (license JWTs aren't checked in).
- The fetcher's 403 handling surfaces as a logged error, not a crash. Cached packs are tier-correct.

## Open questions to resolve during the test

- Does the license server set `tier` as exactly `"developer"` / `"scale"` / `"enterprise"`, or as internal enum values? If enums, the CloudFront Function comparator + the `LicenseValidator.check()` claim-extraction must match the producer's serialization.
- Does the CDN cache 403 responses? Probably not desired — a customer who upgrades should get 200 immediately. If the CDN caches, add `Cache-Control: no-cache` to the function's 403 response.
- Does the OSS `LicenseValidator` 24-hour in-memory cache mask tier changes during the test? Use `force=True` on every refresh to bypass.
