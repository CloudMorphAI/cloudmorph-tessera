# v0.3.0 — Complete OAuth 2.1 Resource Server Implementation

## Problem

What v0.2.0 ships under `tessera/auth/` is a JWT validator — three authenticator classes (`BearerTokenAuthenticator`, `JWTAuthenticator`, `OIDCAuthenticator`) that consume incoming Bearer JWTs, verify them against a JWKS endpoint, and produce an `AuthContext`. Sufficient for the agent-already-has-a-JWT case.

What's **not** present is a full OAuth 2.1 Resource Server. Specifically, the OSS package does not implement:

- **Dynamic Client Registration (RFC 7591)** — no `POST /oauth/register` endpoint where agents can register themselves and obtain credentials.
- **Token introspection (RFC 7662)** — no `POST /oauth/introspect` endpoint that returns metadata about a given token.
- **PKCE handling (RFC 7636) for grant-side flows** — PKCE today is the agent client's responsibility against the upstream IdP; Tessera doesn't participate.
- **`.well-known/oauth-protected-resource` metadata (RFC 9728)** — no discovery document advertising Tessera as a protected resource with its accepted scopes, issuers, and audiences.
- **JWKS publishing by Tessera itself** — Tessera consumes external JWKS but does not publish one for tokens it issues. Today it issues no tokens; if it ever needs to (e.g., signed receipts), the JWKS endpoint would be required.

Enterprise procurement conversations have started asking "does Tessera implement OAuth 2.1 as a Resource Server, or only as a token consumer?" The answer today is "token consumer only." Closing that gap unblocks enterprise sales and aligns the surface with the emerging MCP-spec auth direction (the MCP working group has signaled OAuth 2.1 + DCR as the preferred agent-auth path).

## Scope

The work is bounded to additive endpoints + a metadata document; no breaking changes to the existing `BearerTokenAuthenticator` / `JWTAuthenticator` paths.

### 1. Metadata document

Serve `GET /.well-known/oauth-protected-resource` returning the standard RFC 9728 metadata:

```jsonc
{
  "resource": "https://tessera.example.com",
  "authorization_servers": ["<configured IdP issuer URL(s)>"],
  "scopes_supported": ["tessera:proxy", "tessera:admin", "tessera:audit:read"],
  "bearer_methods_supported": ["header"],
  "resource_documentation": "https://cloudmorph.ai/docs/tessera/auth"
}
```

The `authorization_servers` list comes from `cfg.auth.jwt.issuer` (when `auth.type: jwt`) or `cfg.auth.management_plane.issuer` (when management-plane is wired). Scope names are TBD during implementation — verify against what the proxy actually checks today (likely just the existence of a valid scope claim, not specific values).

### 2. Dynamic Client Registration endpoint

`POST /oauth/register` accepts an RFC 7591 client metadata document and either:

- **Proxies to a configured upstream registration endpoint** (the typical case — DCR is an authorization server feature, not a resource server feature; Tessera as a Resource Server doesn't typically host DCR itself). Configure under `auth.dcr.upstream_url`.
- **Or rejects with 501** when no upstream is configured, with a clear error message pointing to the standard pattern.

Most enterprise IdPs (Okta, Auth0, Keycloak) implement DCR natively; the Resource Server's role is to advertise its scopes and discoverability, not to act as a token issuer. The endpoint exists for spec completeness and to let agents bootstrap when their IdP is unknown.

### 3. Token introspection endpoint

`POST /oauth/introspect` accepts a token + client credentials, returns RFC 7662 introspection response:

```jsonc
{
  "active": true,
  "scope": "tessera:proxy",
  "client_id": "<from JWT claim>",
  "username": "<from JWT sub>",
  "exp": <unix epoch>,
  "iss": "<configured issuer>",
  "aud": "<configured audience>"
}
```

The endpoint is rate-limited (introspection is a common attack vector; brute-force introspection on a series of guessed tokens leaks active-token state). Default: 100 requests/minute per source IP, configurable.

Introspection in v0.3.0 returns claims from the cached JWT validation; it does not call out to the upstream IdP for revocation status. Revocation-aware introspection would require calling the IdP's introspection endpoint — out of scope for v0.3.0 and gated on whether enterprise sales actually need it.

### 4. PKCE pass-through

PKCE-aware OAuth flows are agent ↔ IdP, not agent ↔ Tessera. Tessera's role is to verify the resulting access token has the right claims. The work here is documentation, not code: make the auth section of `docs/CONFIGURATION.md` explicit that PKCE is the agent's responsibility and that Tessera's `JWTAuthenticator` will accept any valid JWT regardless of which OAuth flow produced it.

If a future enterprise customer needs Tessera-side PKCE handling (e.g., Tessera acts as a confidential client to a third-party API), that's a different architectural addition out of v0.3.0 scope.

### 5. JWKS endpoint (optional / future)

`GET /.well-known/jwks.json` returning Tessera's signing keys. Empty list in v0.3.0 because Tessera does not currently issue tokens. The endpoint exists for forward-compatibility and to satisfy strict OAuth 2.1 validators that demand the discovery surface even when empty.

## Audit considerations

Every `/oauth/*` endpoint hit emits an audit event under the deployment scope (not a per-tenant scope, since pre-auth events don't have one). Event types: `oauth_register`, `oauth_introspect`, `oauth_metadata_fetch`. The audit chain integrity is preserved across these events via the same `HashChain` mechanism.

## Acceptance criteria

- `GET /.well-known/oauth-protected-resource` returns valid RFC 9728 metadata.
- `POST /oauth/introspect` with a valid Bearer JWT returns active=true with correct claims.
- `POST /oauth/introspect` with an expired or invalid JWT returns active=false.
- `POST /oauth/register` either proxies to a configured upstream or returns a 501 with a clear message.
- `tests/test_oauth_resource_server.py` covers the metadata document, introspection happy/sad paths, and DCR proxy behavior.
- An OAuth-spec-compliance validator (e.g., the OAuth Working Group's `oauth-toolkit` test harness, or a hand-rolled smoke test against the spec sections) confirms the resource server passes.

## Out of scope

- **Acting as an authorization server.** Tessera does not become a token issuer; it remains a Resource Server. Customers continue to use Clerk/Auth0/Cognito/Keycloak for token issuance.
- **Revocation-aware introspection.** Today's cached JWT validation is sufficient for the v0.3.0 sales conversation; revocation lookup against the IdP can ship later if needed.
- **OAuth 2.0 (1.0) legacy compatibility.** The implementation targets RFC 6749 + the OAuth 2.1 consolidated draft only.
